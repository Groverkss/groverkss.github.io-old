Introduction

Domain Specific Languages: A computer language specialized to a particular application domain.

Paths of a language

The network of paths of an implementation represented as a mountain:

You start off at the bottom with the program as raw source text, a string of characters. Each phase analyzes the program and transforms it to some higher-level representation where the semantics (what the author wants the computer to do) become more apparent.

At the peak, we have a bird's eye view of the user's program and can see what the code means. We begin our descent down the other side of the mountain, transforming the highest-level representation down to successively lower-level forms to get closer and closer to something we know how to make the CPU actually execute.

Starting with source code:

Scanning

Scanning, aka Lexing, aka (incase trying to impress your crush) lexical analysis.

A scanner (or lexer) takes in a linear stream of characters, and groups them together into chunks called tokens. White space is usually insignificant and often discarded by the scanner.

Output from this phase:

Parsing

This is where our syntax gets a grammer. The ability to use the tokens to compose larger expressions.

A parser takes the flat sequence of tokens and builds a tree structure that mirrors the nested nature of the grammer. These trees have a couple of different names — parse tree or abstract syntax tree — depending on how close to the bare syntactic structure of the source language they are. In practice, they are called syntax trees, ASTs.

Parser's job is alose to report syntax errors.

Parsing and Scanning, as expected are similar across all implementations.

Static Analysis

The first part of analysis is called binding or resolution. For each indetifier, we find out where that name is defined and wire the two together. This is where scope comes into play — the region of source code where a certain declaration can be found.

If the language is statically typed, this is where we type check. We report type error if any errors.

This part of the implementation is considered the frontend. Extracting information out of the language.

Storing this information

Places where we can store this information:

  1. Attributes in the syntax tree.
  1. Additional symbol tree with all attributes for each identifier.
  1. Transform the tree into an entirely new data structure that more directly expresses the semantics of code.

We now move on to middle end part.

Intermediate representations

The frontend of the pipeline is specific to the source language the rpogram is written in. The backend is concerned with the final architecture where the program will run.

In the middle, the code may be stored as some intermediate representation (IR) that isn't tightly tied to either the source or destination forms. The IR acts as an interface between two languages.

The IR allows to support multiple architectures by writing a single frontend for the language to convert it to the IR and multiple backends for each architecture.

Optimization

Once we extract the semantics of the program, we swap it with a different program that has the same semantics but implements them more efficently. This is called optimization.

Code generation

This part is concerned with converting the IR after applying all optimizationsto something the machine can run. This part is the backend of implementation.

We have to make a decision here. Do we generate instructions for a real CPU or a virtual one.

Generating for a real CPU makes the native code fast but not portable. The counter this, we can also generate bytecode (because each instruction is usually one byte long).

The synthetic instructions are designed to be less tied to the peculiarities of any one computer architecture. These can be thought of as dense, binary encoding of the language's low-level operations.

Virtual Machine

This phase only comes if bytecode is generated.

To run the bytecode, we can either write a mini-compiler to convert byte code to machine code for each architecture or you can write a virtual machien (VM) that emulates a hypothetical chip supporting your virtual architecture at runtime. The VM simulates each instruction at runtime.

Runtime

If compiled to machine code, we can just run the code on the machine. Otherwise, we start the virtual machine and load the program on it.

There also goes other kind of stuff during runtime, for example: Garbage collection.

Shortcuts and Alternative Routes

Single-pass compilers

Combine parsing, analysis and code generation into a single step. These single-pass compilers restrict the design of the language. No global information about the program, and you don't revisit parsed parts of code. This means as soon as you see an expression, you need to know enough to correctly compile it.

Pascal and C were designed with this limitation due to less memory in old systems. This is the reason C cannot call a function unless declared before.

Tree-walk Interpreters

Some programming languages begin executing code right after parsing it to an AST.

Transpilers

You write a frontend for your language to target an existing language and use existing compilation tools to compile the language.

These are called source-to-source compilers or a transcompiler.

Just-in-time compilation

This is used when you do not know your end user's architecture. This kind of compiler compiles just when the program is loaded.

More sophisticated JITs insert profiling hooks into generated code to see performance critical code and what kind of data is flowing through them. Then, over time, they will automatically recompile those hot spots with more advanced optimizations. (HostSpot JVM).

Compilers and Interpreters

  • Compiling→ implementation technique that involves translating a source language to some other — usually lower-level — form. When generating bytecode or machine code, you are compiling. When you transpile to another higher language, you are compiling.
  • Language implementation is compiler → Translates source code to some other form but doesn't execute it.
  • Language implementation is interpreter → Takes in source code and executes immediately.
