Introduction

Property

Core: A set of operating systems communicating over networks to get a task done.

If a task can be done with a single computer, do it that way. Use distributed systems as a last resort.

Why people build this stuff?

  • Parallelism
  • Fault Tolerance: If one of them breaks, still keep running.
  • Physical Reasons: Systems that are inherently distributed.
  • Security: Run some code that you may not trust. Don't want it to interact with lets say my private server. Isolating something.

Most of the course on 1. and 2.

Challenges:

  • Concurrency
  • Partial Failure
  • Performance

Infrastructures:

  • Storage: most focused and fairly straightforward
  • Communication: not the focus here
  • Computation: things like map-reduce

Goal:

An interface/abstraction that acts as if the underlying distributed system is just a regular system. How do you achieve that? Implementation.

example:

  • RPC (remote procedure call) masks that we are communicating over unreliable network
  • threads: way of structuring concurrent operations
  • concurrency control - locks

Performance goals:

  • Scalability: Example → 2 ×\times×﻿ computer ===﻿ 2 ×\times×﻿ throughput
  • Fault tolerance:
    • Core requirements:
      • Availability: If some system goes down, it still continues operating correctly.
      • Recoverability: If a system went down and is back up again, it can join the system back by syncing with rest of the server.
    • Tools used most commonly
      • Non-volatile storage: Less updates
      • Management of Replication: Two replications of the system go out of sync
  • Consistency:
    • Strong: Guarantee to get the latest value from a DB
    • Weak: Hold no such guarantee.
