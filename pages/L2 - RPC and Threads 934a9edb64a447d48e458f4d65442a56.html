Go advantages: Threads, garbage collected so easier to handle memory

Goroutines = threads

Why use Modern I/O Concurrency / Threads:

  1. One program calls many RPCs to interface with other systems/programs and waits for communication. While one activity waits others can proceed.
  1. Parallelism (computing one program faster by splitting into threads) is of less importance to this course.
  1. Convenience programs like periodic checks, send heartbeat messages can be implemented as separate threads.

Alternate: Event driven programming where a single thread waits for some event inputs to execute the corresponding code.

It's easier to write sequential code in a thread than to chop up activity into pieces that can be activated based. But threads also have some overheads, which don't exist in event-driven programs so sometimes the extra programmer effort might be needed.

Why can't we add locks within the Map datastructure itself inbuilt?

a) Locking has overheads which might be unnecessary for most programs so it's a bad idea to have it running all the time inbuilt, let the programmer decide instead.

b) It can lead to deadlocks if the programmer is not careful then, but they don't know how locks are used inside the map so they will mess up.

Goal of RPCs is to behave like a normal procedure call on the same program.

When the client calls an RPC, a stub packs it into the function name, arguments, etc. and this package is passed over the network and received by the server's stub which unpacks the package and calls the function required on the server. It then sends back the returned value as a package to the client stub which is waiting, and when it gets the values it sends it to the caller function on the client.

Go's RPC system is atmost-once under failure, there will be no duplication but sometimes it won't run even once and the client will be returned the error. It is the client application's responsibility to retry (or maybe not).
